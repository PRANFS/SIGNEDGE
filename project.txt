================================================================================
SignEdge – Project Architecture Overview
Group 29 – INF2009 Edge Computing & Analytics
================================================================================

Project Goal
------------
Privacy-first, fully offline, sub-100 ms latency edge device running on Raspberry Pi 5
for real-time bidirectional communication aid between Deaf and hearing people in
high-stakes environments (hospitals, banks, etc.).

Two main translation directions:
1. Hearing-to-Deaf: Spoken English → displayed text
2. Deaf-to-Hearing: ASL finger-spelling (static A–Z letters) → spoken English

Core Constraints & Design Principles
------------------------------------
- End-to-end latency target: < 100 ms (measured loop ~51 ms)
- Strict privacy: raw video + raw audio NEVER leave the device; only final text is output
- Offline-first / no cloud dependency → reliable in network-dead zones or outages
- Runs entirely on-device (no internet required)
- Hardware must stay within Raspberry Pi 5 (8 GB RAM) thermal & resource limits

Hardware Platform
----------------
To ensure that the system meets the strict real-time (<100 ms), offline, and privacy-first constraints of the project, we conducted a detailed 
evaluation of multiple edge hardware platforms before finalizing our choice.

Evaluation Criteria
The hardware selection was guided by the following requirements:
• Sufficient RAM for concurrent audio + vision pipelines 
• Ability to sustain real-time inference workloads
• Stability under continuous operation
• Mature software ecosystem with minimal integration risk
• Compatibility with fully offline deployment

=======================================================

Evaluated Options
1. Raspberry Pi Zero 2 W

Limitations Identified:
• Severe RAM Constraints (512 MB total)
After OS, drivers, and runtime initialization, very limited memory remains

Insufficient headroom for:
    • Model loading
    • Frame buffering
    • Continuous inference

Performance Bottlenecks
•   Real-time vision workloads operate under heavy memory pressure
•   Estimated performance: ~1–3 FPS
•   Not suitable for sub-100 ms latency target


System Stability Risks
•   High likelihood of crashes or slowdowns during prolonged operation
•   No scalability for future feature expansion

Conclusion:
Not viable for reliable, real-time edge AI workloads.

=======================================================
2. Milk-V Duo (with NPU)
Advantages Considered:
•   Integrated NPU with promising theoretical performance
•   Low-power edge AI design

Critical Constraints Identified:
•   Memory Limitations (512 MB configuration)
•   Estimated application requirement: 150–200 MB

After OS load → minimal remaining headroom

High risk of:
•   Runtime instability
•   Memory exhaustion during inference

Immature Software Ecosystem
•   Requires model conversion to proprietary SOPHON format
•   Limited documentation and tooling support
•   Incomplete operator support for common ML layers

Development Risk
•   Unsupported layers fall back to CPU execution
•   Increased debugging complexity
•   Reduced real-world performance vs theoretical NPU capability

Conclusion:
High integration risk and unsuitable within the project’s limited timeline.
=======================================================
Final Decision: Raspberry Pi 5 (8 GB RAM)
The Raspberry Pi 5 was selected as the primary edge device due to its superior balance of performance, reliability, and ecosystem maturity.

Key Advantages
•   Significantly Higher Memory Capacity
8 GB RAM provides ample headroom for: OS + runtime
Vision pipeline (MediaPipe + MLP)
Audio pipeline (speech-to-text model)
Eliminates memory pressure seen in alternative platforms

Stable Real-Time Performance
•   Capable of sustaining:
~30–40 ms vision loop
~51 ms total system latency
Meets sub-100 ms project requirement

Mature Software Ecosystem
•   Extensive community support and documentation
•   Native compatibility with:
    MediaPipe
    Python ML stack

Audio processing libraries
•   No need for proprietary model conversion pipelines

Low Development Risk
•   Rapid prototyping and debugging
•   Predictable performance characteristics
•   Strong tooling support

Scalability
•   Supports future enhancements such as:
    Larger models, Multi-modal fusion improvements, Additional sensors or features

Optional Hardware Enhancements
To further optimize system performance:

•     Hailo AI Accelerator (if available)
Offloads inference workload from CPU
Improves throughput and latency consistency

•   Ribbon Cable Camera Module (CSI)
Lower latency compared to USB webcams
Reduced CPU overhead
Dedicated bandwidth and more stable frame timing

Software Architecture & Data Flow
-----------------
Central node: Raspberry Pi 5 (8 GB RAM variant)
Inputs:
  • RGB webcam (720p @ 30 fps recommended) – captures hand gestures
  • Microphone – captures spoken English
Outputs:
  • HDMI display / screen – shows transcribed text (hearing → Deaf)
  • Integrated or connected speakers – plays synthesized speech (Deaf → hearing)

Mandatory: Active cooling (fan/heatsink) to prevent thermal throttling during continuous inference

Software Architecture & Data Flow
---------------------------------
All processing is local. High-level layers:

[INPUT LAYER]                  → [EDGE PROCESSING UNIT (RPi 5)]                  → [OUTPUT LAYER]

Inputs ─┬─ Webcam (visual – ASL finger spelling)                                 ┌─ HDMI Display (text)
        └─ Microphone (audio – spoken English)                                  └─ Speakers (audio)

                          │
                          ▼
                Sensor Fusion / State Manager
                          │
           ┌──────────────┼──────────────┐
           │                             │
   Visual Pipeline                Audio Pipeline
 (Deaf → Hearing)             (Hearing → Deaf)
           │                             │
           ▼                             ▼
   MediaPipe Hands               Lightweight offline
   → extracts 21 hand            Speech-to-Text engine
     landmarks (x,y,z coords)         (local model)
   → raw pixels discarded
     immediately for privacy
           │
           ▼
   Tiny MLP Classifier
   (multi-layer perceptron)
   → classifies static shape
     → one of 26 ASL letters (A–Z) or space or del
           │
           ▼
      Post-processing
      (smoothing, confidence filtering, etc.)
           │
           ▼
     Text / Command
           │
           ▼
     Text-to-Speech
           │
           ▼
        Output

Important implementation notes:
• Speech-to-Text runs in a separate thread → does not block the visual inference loop
• MediaPipe landmark extraction: ~15 ms
• MLP classification: ~10 ms
• Total vision loop (capture → detect → classify → smooth): ~30–40 ms
• Overall system loop: ~51 ms (well under 100 ms target)

Model & Approach Justification
------------------------------
Vision side (ASL finger spelling):
  Selected: MediaPipe Hands + tiny MLP on 21 landmarks only
    → Pros: ~25–35 ms total, very lightweight, discards raw pixels immediately (privacy), fits easily in RAM
  Rejected:
    • YOLOv8 → too heavy, >200 ms latency on RPi 5 CPU
    • 3D-CNN / ResNet video classification → excessive RAM (frame buffering), unnecessary for static shapes

Audio side:
  Lightweight, offline speech-to-text model (exact engine not specified in docs, but ~1 GB RAM consumer)

Resource Budget (approximate)
----------------------------
• Total estimated RAM usage: ~1.6 GB
  - Speech model: ~1 GB (biggest consumer)
  - OS + visual pipeline + buffers: remaining ~0.6 GB
  → comfortable margin inside 8 GB
• CPU allocation strategy:
  - 2 cores: OS, UI, I/O handling
  - 2 cores: pinned for real-time inference (minimize interruptions)
• Thermal: Active cooling mandatory + "turn-taking" logic to allow CPU cooldown

Risks & Mitigations (selected)
------------------------------
• Thermal throttling → active cooling + turn-taking scheduling
• Privacy leakage → strict offline design, no storage/transmission, raw data discarded
• Poor detection (lighting, framing) → MediaPipe skips bad frames; after 3 skips → user prompt
